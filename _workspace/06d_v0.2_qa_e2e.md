## 요약

v0.1.0의 스크립트 한 개짜리 placeholder(`scripts/qa/run-all.ts`)를, 실 Postgres + 실 Prisma + 실 NestJS Service 를 직접 호출하는 vitest 기반 통합 테스트 스위트로 확장했다.

- **테스트 러너:** vitest (NestJS 앱 내부)
- **격리 전략:** 단일 fork(`pool: 'forks'`, `singleFork: true`) + describe 단위 TRUNCATE
- **Mock 범위:** PortOne 외부 HTTP만 stub. DB·암호화·Referral 엔진·UserService 가드는 전부 실 구현 호출
- **커버리지:** 기준 시나리오(1,000,000원) + 반올림 edge + 환불 역정산(전체/부분/late) + 어뷰징 6종(A1-direct/A1-ancestor/A2/A3/T5/T6) + Boundary shape(7 스키마) + 결제 confirm 경로 6종

## 실행 방법

### 1. 실행 전 필요 조건

```bash
# (1) Postgres 기동 (infra-engineer가 준비 중인 docker-compose.yml 기준)
docker compose up -d postgres

# (2) Prisma 마이그레이션 (backend-engineer M1 결과물)
pnpm --filter @nuxia2/api prisma:migrate

# (3) 시드는 테스트에서 TRUNCATE 하므로 필수 아님. 스키마만 있으면 됨.

# (4) 환경변수 (apps/api/.env.test 로 복사)
cp apps/api/.env.test.example apps/api/.env.test
```

### 2. 전체 E2E 실행

```bash
# 권장 (CI 포함)
pnpm --filter @nuxia2/api test:e2e

# 또는 오케스트레이터 경유 (스프린트 스펙상 명령)
pnpm exec tsx scripts/qa/run-all.ts
```

### 3. DB 없이 pre-flight 체크

```bash
pnpm exec tsx scripts/qa/baseline-math.ts
# → "OK — baseline referral math intact (3%/5%/17%)."
```

### 4. 단일 파일만 실행

```bash
pnpm exec vitest run --config apps/api/vitest.config.ts apps/api/test/referral-engine.test.ts
```

### 5. watch 모드

```bash
pnpm --filter @nuxia2/api test:watch
```

## 설계 결정 요약

### (a) Service-level integration vs HTTP e2e

Nest HTTP 서버를 띄우지 않고 서비스 클래스를 **직접 인스턴스화**하여 호출한다. 장점:
- startup 오버헤드 없음 → 수십 개 테스트가 1~2초 안에 끝남
- Nest DI/Pipes 없이도 서비스 간 경계를 명확히 검증 가능
- PortOne 만 stub 하면 나머지는 프로덕션 코드 그대로

단점: Guards/Interceptors/Pipes 경로는 이 스위트로는 검증 안 됨 → 그쪽은 추후 supertest e2e(미래 스프린트)에서 커버.

### (b) Mock 최소화 원칙

- `PortOneClient` 만 `StubPortOneClient` 로 대체 (외부 HTTP)
- `PrismaClient` 는 실 DB 접속
- `crypto.util` 은 실 AES-GCM 동작 — `APP_ENCRYPTION_KEY` 테스트용 기본값 주입
- `referral.distribute` / `user.createUser` 전부 실 구현

### (c) BigInt 비교

zod `BigIntStringSchema` 가 문자열 기반이므로 DB 원장 비교는 `toBe(N_n)` 으로 직접 BigInt 비교, API 응답 DTO는 `.toBe('250000')` 같은 string 비교를 병행.

### (d) 테스트 격리

- `pool: 'forks'` + `singleFork: true` → 테스트끼리 DB 트랜잭션 경합 없음
- 각 describe 블록 시작에서 `await clearAll()` (TRUNCATE CASCADE)
- `createUser()` 는 매번 `uniqueCi()` 로 ci 중복 회피 (같은 describe 안에서)

## 기준 시나리오(1,000,000원) 코드 라인 수

- 약 25 줄 (setup 포함 describe 내 fixture 호출)
