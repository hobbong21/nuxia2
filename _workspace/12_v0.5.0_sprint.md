# 12. v0.5.0 스프린트 스펙

> 작성: 오케스트레이터
> 기반: `CHANGELOG.md [Unreleased] Planned (v0.5)` + 사용자 요청

## 목표

v0.4에서 정의만 한 관측·감사 지표를 **실제 도메인 경로에 연결**하고, 관리자가 **AuditLog를 UI로 조회**하며, 사용자가 **2FA 백업 수단(SMS/이메일)**을 선택할 수 있게 만든다.

## 범위 (Must)

### M1. Metrics 도메인 연결 (backend-engineer)

v0.4에 정의된 5 counter/gauge를 실제 서비스 경로에서 `metrics.inc*()` 호출:

- **Referral Engine** (`referral/engine.service.ts::distribute`):
  - 성공 시 `metrics.incReferralDistribute('success')` × 세대 수
  - 수혜자 없음/차단 상태 건너뜀 시 `.incReferralDistribute('skipped')`
  - 예외 catch 시 `.incReferralDistribute('failed')` 후 rethrow
- **Payment Confirm** (`payment/payment.service.ts::confirm`):
  - 포트원 금액 일치 → 트랜잭션 성공 시 `.incPaymentConfirm('success')`
  - 금액 불일치 throw 시점에 `.incPaymentConfirm('mismatch')`
  - 그 외 실패 `.incPaymentConfirm('failed')`
- **Abuse 차단** (`user/user.service.ts::createUser`의 각 throw 지점):
  - `.incAbuseBlocked('SELF_REFERRAL')` / `'CIRCULAR'` / `'DUPLICATE_CI'` / `'STAFF_REFERRAL_FORBIDDEN'` / `'WITHDRAW_REJOIN_COOLDOWN'`
  - 기존 ThrottlerGuard rate limit → `'RATE_LIMIT'`
- **Webhook** (`webhook/portone.webhook.controller.ts`):
  - 유효 서명 + 신규 이벤트 → `.incWebhookReceived('portone', 'ok')`
  - 중복(duplicate) → `.incWebhookReceived('portone', 'duplicate')`
  - 서명 거부 → `.incWebhookReceived('portone', 'rejected')`
- **Minor Hold Gauge** (`metrics.service.ts::refreshMinorHoldGauge`):
  - 주기적(1분) 또는 on-demand 호출로 `User.status='MINOR_HOLD'` 카운트 → `minorHoldGauge.set(N)`
  - 크론/인터벌은 NestJS `@Interval(60_000)` 데코레이터 (nestjs/schedule 추가)

### M2. Admin UI — AuditLog 조회 화면 (frontend-engineer)

- **BE 신규 엔드포인트** (backend에서 추가):
  - `GET /admin/audit-logs?kind=&actorUserId=&targetType=&targetId=&cursor=&limit=`
  - 응답: `PaginatedAuditLogsSchema` (shared-types)
- **shared-types**:
  - `AuditLogSchema` — id, actorUserId, actorNickname?, kind(enum), targetType, targetId, createdAt, diffSummary(optional)
  - `AuditLogKindSchema` — USER_FLAG / USER_RELEASE_MINOR / USER_MARK_STAFF / USER_SUSPEND / PAYOUT_RUN / PAYOUT_RELEASE / (+ 확장 가능)
- **Frontend 신규 라우트**:
  - `apps/web/app/(admin)/admin/audit-logs/page.tsx` — 테이블 + kind 필터 + actor 검색 + target 필터
  - 컴포넌트 재사용: `DataTable`, `AbuseKindBadge`(→`AuditKindBadge`로 포크), 사이드바 메뉴 추가
  - 각 로그 클릭 시 diff 상세 모달 (actorNickname → target → before/after JSON 뷰)
- **사이드바**: `/admin/audit-logs` 메뉴 아이콘 "📝" 추가

### M3. SMS/이메일 OTP 백업 (backend-engineer + frontend-engineer)

**Backend**:
- Prisma 스키마 확장:
  - `User.phoneE164` (암호화, 기존 포트원 본인인증에서 획득 가능)
  - 새 모델 `OtpChallenge { id, userId, kind(SMS|EMAIL), codeHash, expiresAt, attemptCount, usedAt }`
- `apps/api/src/modules/auth/otp.service.ts`:
  - `requestOtp(userId, kind='SMS'|'EMAIL')` — 6자리 코드 생성, bcrypt 해시 저장, 3분 TTL, 전송 (SMS/이메일 공급자는 mock — 실전 연동은 env 기반 adapter)
  - `verifyOtp(userId, code, kind)` — 해시 검증, 5회 실패 시 lock, 성공 시 `usedAt` 기록
- `/auth/otp/request` + `/auth/otp/verify` 엔드포인트
- 로그인 2단계 대체 경로: TOTP 없거나 lockout 시 "코드를 SMS/이메일로 받기" 옵션
- SMS adapter: `SolapiAdapter` (한국 시장) — dry-run 모드(env `OTP_DRY_RUN=1`) 시 console.log만
- Email adapter: `NodemailerAdapter` — dry-run 동일

**Frontend**:
- `TotpField` 하단에 "SMS/이메일로 받기" 토글 (`components/auth/OtpChannelPicker.tsx`)
- `/mypage/security`에 SMS/이메일 백업 설정 UI (전화번호/이메일 등록 여부 표시)
