## Assumptions (가정)

문서 전체에서 아래 가정이 미확정이며, 향후 사용자·법무·세무 확인이 필요합니다.

1. **본인인증(`ci`) 필수**: 모든 구매 회원은 포트원 본인인증을 통과해야 하며, 본인인증 기관이 반환하는 `ci`(연계정보) 88자리 해시는 1인당 1개로 간주한다. `ci`가 없는 상태로는 주문·레퍼럴 수익 수취 모두 불가.
2. **미성년자(만 19세 미만)는 가입 가능하나 레퍼럴 수익 수취는 불가**. 본인인증에서 DOB 파싱으로 판정하며, 수익은 발생 시점에 `WITHHELD` 상태로 유보.
3. **레퍼럴 수익은 "용역 대가" 또는 "사업소득"**으로 취급한다고 가정하며, 연간 누적 지급액 합산 필요. 원천징수 구현 여부는 MVP Should 범위(세무 자문 대기).
4. **화폐 단위는 원(KRW) 단일**이며, 레퍼럴 배분 시 **원 단위 내림(floor)** 이 기본이고, 내림 후 남는 절삭 잔액은 플랫폼 귀속(= 24.xx% 미만으로 지급될 수 있음).
5. **유보 기간 = 7일**(구매확정 기본 자동확정 주기와 동일). 이 기간 동안 환불·분쟁 발생 시 원장은 생성되지만 `Payout` 대상에서 제외.
6. **결제수수료는 평균 3.0%**(포트원 통합 수수료 가정, 카드·간편결제 가중평균). 정산 마진 계산에만 사용하며 고객에게 전가하지 않음.
7. **단일 통화·단일 결제 건 가정 유효**하나, 본 문서는 분할결제(다회차)·혼합결제(현금+포인트) 대비 정책을 포함.
8. **정산 주기**: 월 1회, 매월 1일 ~ 말일분을 익월 10일에 지급. 계좌 등록 미완료 수취인은 지급 보류(`WITHHELD`).
9. **3세대 초과(4대 이상)**에는 어떠한 금액도 귀속되지 않는다. 상위 추천인이 3명 미만이면 해당 세대는 플랫폼 귀속.
10. **환불·취소 가능 기간 = 구매확정 전 + 구매확정 후 7일(청약철회)**. 이후 분쟁은 개별 CS 처리이며 레퍼럴 역정산은 원칙적으로 수행하지 않음(정책적 컷오프).
11. **관리자·CS 직원 본인 및 직계 계정은 레퍼럴 체인 참여 불가**(내부 어뷰징 방지).
12. **추천 관계는 가입 시 1회 확정**되고 이후 변경 불가(체인 이관 기능 없음).

## 기능 스코프 (MoSCoW)

### Must (MVP 필수)
- 회원가입·로그인(이메일+비밀번호, 소셜 로그인 1종 이상, 포트원 본인인증)
- 추천인 코드(`referralCode`) 기반 가입 플로우(URL 파라미터 자동 주입 포함)
- 상품 목록/상세/검색(키워드+카테고리), 카트, 주문, 배송지 관리
- 포트원 V2 결제위젯 연동(카드·간편결제·가상계좌)
- 주문 상태 기계(`PENDING_PAYMENT → PAID → PREPARING → SHIPPED → DELIVERED → CONFIRMED`)
- **3세대 레퍼럴 원장 생성** (구매확정 시, 1대 3% / 2대 5% / 3대 17%)
- **환불/취소 역정산 원장 생성**
- 어뷰징 방지: 셀프레퍼럴·순환참조·1`ci`=1계정·IP/디바이스 지문 기록
- 정산(월 1회) 관리자 대시보드 기본 CRUD(수동 승인 허용)
- 리뷰 작성(주문 확정자에 한함), 별점+텍스트
- 모바일 반응형 웹 + Capacitor 빌드(iOS/Android)
- 한국어 i18n, KRW 표시 규칙, 통신판매업·개인정보처리방침 링크

### Should (강화)
- 포트원 본인인증 결과로 19세 이상·성별·생년월일 유보 판정

- 관리자용 어뷰징 콘솔(플래그 계정 수동 심사 UI)

- 포트원 간편결제 결제 실패 재시도 플로우
- 푸시 알림(Capacitor + FCM/APNs) — 주문 상태, 정산 확정
- 쿠폰·프로모션(정액/정률), 쿠폰과 레퍼럴 정산 기준액 분리 규칙
- 부분 환불(품목 단위), 다건 환불

### Could (있으면 좋음)

### Won't (본 릴리스 제외)

### 시나리오 스케치 (페르소나별)

- **P1 정상 플로우**: SNS에서 추천 링크 클릭 → 상품 상세 → 카트 → 본인인증(최초 1회) → 포트원 결제위젯 → 완료 콜백 → 주문 내역.
- **P1 실패 플로우**: 결제 중 네트워크 끊김 → 앱 재진입 시 `paymentId` 상태 동기화 API로 "결제 진행 중" 복구 안내.
- **P2 정상 플로우**: 리퍼럴 대시보드 → 이번 달 예상 수익 확인 → 유보 해제 일자 확인 → 정산일 후 입금 확인.
- **P2 실패 플로우**: 하위 구매자가 환불 → 대시보드에 `REVERT` 원장이 마이너스로 표시되고 사유 뱃지 노출.
- **P3 정상 플로우**: 어뷰징 콘솔 → 점수 ≥ 임계치 계정 목록 → 근거 확인(IP/디바이스/주문 패턴) → `suspend` 처리.
- **P3 실패 플로우**: 이미 정산된 계정이 사후 어뷰징 판정 → 지급된 금액은 환수 청구 기록만 남기고 추가 지급 정지.

## 레퍼럴 정책 상세표

### 0. 비율 정의 (단일 진실)

| 세대 | 관계 설명 | 비율 |
|------|----------|------|
| 1대 | 구매자의 직접 추천인 | **3%** |
| 2대 | 1대의 추천인 | **5%** |
| 3대 | 2대의 추천인 | **17%** |
| 합계 | — | **25%** |

**기준 검증**: 3대 구조(A→B→C→D)에서 D가 **1,000,000원** 결제·확정 시 →
- 1대 C: 1,000,000 × 0.03 = **30,000원**
- 2대 B: 1,000,000 × 0.05 = **50,000원**
- 3대 A: 1,000,000 × 0.17 = **170,000원**
- 합계: **250,000원** (= 25%)

### 1. 정상 케이스 (구매확정 후 정산)

| 항목 | 값 |
|------|---|
| 기준 금액 | 주문의 **상품 실결제 금액** (배송비·쿠폰 차감 후). 포인트 사용분은 기준에서 **제외**. |
| 계산 단위 | 원 단위 `floor`. 예: 999원 × 3% = 29.97 → **29원**. |
| 원장 생성 시점 | **구매확정(Confirmation) 이벤트** 또는 구매확정 후 자동 확정(주문 완료 +7일). |
| 유보 기간 | 구매확정 후 **7일**. 이 기간 내 환불 시 `REVERT` 처리. |
| 지급 시점 | 유보 종료 후 다음 정산 주기(월 10일). |
| 상위 결손 | 1대만 존재(2대 없음) → 2·3대 금액은 **플랫폼 귀속**(생성 안 함). |
| 본인 추천 체인 차단 | 구매자와 `ci`가 같은 체인은 원장 생성 전 거부. |

### 2. 환불/취소/혼합결제 6대 케이스

| # | 케이스 | 트리거 이벤트 | ReferralLedger 액션 | Payout 영향 | 비고 |
|---|-------|--------------|--------------------|--------------|------|
| **C1** | 정상 구매확정 | `order.confirmed` | 최대 3개 `EARN` 원장 생성 (1대 3%, 2대 5%, 3대 17%) | 유보 7일 후 지급 | 기준 시나리오 |
| **C2** | 전체 환불 (구매확정 전) | `order.refunded.full` (status=PAID) | 원장 미생성 → 생성된 적 없으므로 액션 없음 | 영향 없음 | 결제 전/완료 직후 환불 포함 |
| **C3** | 전체 환불 (구매확정 후, 유보 내) | `order.refunded.full` | 각 `EARN`에 대해 동일 금액의 `REVERT` 원장 생성 (`net = 0`) | `WITHHELD` 해제되지 않음 → 지급 없음 | 유보 기간 중 안전 |
| **C4** | 부분 환불 | `order.refunded.partial(ratio=r)` | 각 `EARN`에 대해 `REVERT` 금액 = `floor(EARN × r)` | 남은 금액만 지급 대상 | 복수 품목 환불 시 각 품목 가액 기준 재계산 |
| **C5** | 구매확정 후 유보 종료 이후 환불 | `order.refunded.lateRefund` | **정책적 역정산 생략**(어뷰징 차단 또는 CS 재량) 원칙, 단 관리자 수동 `REVERT` 가능 | 이미 지급된 경우 환수 대상 → `Payout.status=CLAWBACK_REQUESTED` | 컷오프는 7일 이후 |
| **C6** | 분할/다회차 결제 | 여러 `paymentId` 로 분산 결제 | 각 `paymentId` 별로 확정 시 해당 부분 금액 × 비율로 `EARN` 원장 1세트 생성 | 부분 결제 미완 주문은 원장 생성 보류 | 예: 50만 + 50만 = 100만원 |
| **C7 (보조)** | 혼합 결제(카드 + 포인트) | 단일 주문에 `paymentAmount` + `pointUsedAmount` | **레퍼럴 기준액 = `paymentAmount`만** (포인트 제외) | — | 포인트는 자사 부채, 수수료 없음 |
| **C8 (보조)** | 결제 취소 (승인 후 매입 전) | `payment.cancelled` | 원장 미생성 (구매확정 이전) | 영향 없음 | 대부분 즉시 취소 |

> **원자성 규약**: 동일 주문의 환불은 항상 `EARN`과 `REVERT`를 **같은 DB 트랜잭션**으로 묶어 기록한다. 부분 환불이라도 `net(EARN + REVERT) ≥ 0`을 보장.

### 3. 수치 반올림·유보·세금 정책

- **반올림**: 모든 배분액은 `Math.floor(amount * rate)` 원 단위. 반올림 오차는 플랫폼 귀속.
- **유보 기간**: 구매확정일 + 7일 00:00 KST.
- **정산 주기**: 매월 1일 ~ 말일 확정분 → 익월 10일 (영업일 아니면 다음 영업일) 지급.
- **최소 지급 단위**: 10,000원 이상. 미달 시 차월 이월.
- **원천징수(가정)**: 지급 총액의 3.3%(소득세 3% + 지방소득세 0.3%) 원천징수, 지급 전 차감(관리자 설정으로 ON/OFF 가능; 세무 자문 대기 — TODO).
- **연간 신고**: 사업자 수취인이면 세금계산서 발급 경로 제공, 개인 수취인이면 지급명세서 제출 대상으로 분류(스프레드시트 export 지원).

## 어뷰징 시나리오 & 방지 요구사항

> 모든 시나리오는 **정책 → 검출 → 차단** 3층으로 분리해 기술. 각 이벤트는 `AbuseLog` 엔터티에 구조화 기록.

### A1. 셀프레퍼럴 (본인이 본인을 추천)

- **정책**: 구매자와 추천 체인의 1·2·3대 중 하나라도 `ci`가 일치하면 원장 생성 금지. 가입 시 추천인 `ci`와 신규 회원 `ci`가 일치하면 가입 자체를 차단.
- **검출**:
  1. 회원가입 트랜잭션 내에서 추천인 `user.ci == 신규.ci` 실시간 비교.
  2. 주문 확정 직전 ledger 생성 프리체크에서 `beneficiary.ci == buyer.ci` 재확인.
  3. IP + 디바이스 지문이 동일하고 `ci`만 다른 경우 "의심" 플래그(차단은 아님).
- **차단**:
  - 가입 시: HTTP 400, 에러 코드 `REFERRAL_SELF_FORBIDDEN`, 어뷰징 로그 기록.
  - 주문 시: 원장 생성 스킵, 플랫폼 귀속 처리, 감사 로그 남김.

### A2. 순환참조 (A → B → C → A)

- **정책**: 신규 회원의 `referrerId` 가 본인을 포함한 선조(ancestor)에 나타나면 금지. 이미 존재하는 체인이 순환이 되는 변경은 원천적으로 불가(추천 관계 수정 불가).
- **검출**: 가입 시 추천인 체인을 최대 3단계(정책상 의미 있는 최대) 탐색하며 DFS 하여 자기 자신 감지. N+1 탐색 방지를 위해 ancestor 캐시 컬럼 `ancestorPath text[]` 유지.
- **차단**: 트랜잭션 롤백, HTTP 409, 에러 코드 `REFERRAL_CYCLE_DETECTED`, 해당 요청 IP/디바이스 지문 기록.

### A3. 다중계정 (동일인이 계정 N개)

- **정책**: **1 `ci` = 1 활성 계정**. 동일 `ci` 재가입 시도는 "복구 플로우"로 안내. 탈퇴 후 재가입은 동일 `ci`로 기존 계정 재활성화.
- **검출**:
  1. `User.ci` UNIQUE 제약(DB 레벨).
  2. 디바이스 지문(`deviceFingerprint` = canvas + UA + 타임존 해시) + IP 클래스 C 서브넷 매칭으로 "그룹 점수" 산출.
  3. 동일 결제수단(BIN + 마스킹 카드번호 뒤 4자리 해시)이 서로 다른 `ci` 에 반복 사용되면 점수 가산.
- **차단**:
  - DB 유일성 위배 → HTTP 409, `USER_CI_DUPLICATE`.
  - 그룹 점수 ≥ 70: 추가 주문·레퍼럴 원장 생성 보류, 관리자 수동 심사 큐 진입.
  - 확정 시: 모든 관련 계정 `SUSPENDED` + 미지급 Payout `WITHHELD`.

### A4. 봇 구매 / 자동화 어뷰징

- **정책**: 구매는 본인인증 완료 계정만 허용. 단일 세션 내 1분당 주문 ≤ 3건, 단일 계정 1시간당 주문 ≤ 10건.
- **검출**:
  - 레이트 리밋(Redis): `order:rate:{userId}`, `order:rate:{ip}`.
  - 행동 휴리스틱: 결제위젯 첫 노출부터 승인 요청까지 < 2초(사람 평균 12~30초)이면 점수 +20.
  - 동일 카드/BIN 의 주문이 서로 다른 `ci` 에 5회 이상 반복 시 점수 +30.
- **차단**:
  - 레이트 리밋 초과 시 HTTP 429 + CAPTCHA(웹은 hCaptcha, 앱은 Play Integrity / App Attest).
  - 점수 ≥ 80: 주문 `HOLD` 상태로 전이, 관리자 수동 심사 후 승인/취소 결정.

### A5. (보조) 보상 헌팅 — 리더 지인 교환 구매

- **정책**: 동일 출하지/결제수단/IP가 상호 교환되는 주문은 수익 지급 보류.
- **검출**: 7일 윈도우 내 `buyer → beneficiary` 관계가 양방향으로 성립하는 주문 2건 이상.
- **차단**: 해당 원장 `status = SUSPENDED_FOR_REVIEW`, Payout 제외.

> `AbuseLog` 항목: `type(SELF|CYCLE|MULTI|BOT|SWAP)`, `severity(1~5)`, `detectedAt`, `relatedUserIds[]`, `evidence jsonb`.

## 마진 검증 간단 모델 (BEP)

### 변수
- `R` = 매출 (고객 실결제 현금/카드 금액, 포인트 제외)
- `C_g` = 상품원가 비율 (%)
- `C_r` = 레퍼럴 지급 비율 = **25%** (기준 시나리오, 상위 세대 결손 시 그 이하)
- `C_p` = 결제수수료 비율 = **3%** (가정)
- `C_op` = 기타 운영비 비율 (물류, CS, 인프라)
- `M` = 플랫폼 마진 (세전)

### 기본 식
```
R = C_g·R + C_r·R + C_p·R + C_op·R + M
M/R = 1 - C_g - 0.25 - 0.03 - C_op
    = 0.72 - C_g - C_op
```

### 손익분기 조건 (BEP)
```
M ≥ 0  ⇔  C_g + C_op ≤ 0.72
```

### 시나리오
| 시나리오 | C_g | C_op | 결과 M/R |
|---------|-----|------|---------|
| 고마진(디지털·자체PB) | 0.30 | 0.10 | **+0.32** (수익) |
| 중간(의류 일반) | 0.50 | 0.10 | **+0.12** (수익) |
| 박리다매 | 0.60 | 0.12 | **+0.00** (BEP) |
| 저마진 위험 | 0.65 | 0.12 | **-0.05** (적자) |

### 시사점
- 레퍼럴 25% 전액 지급을 유지하려면 **원가율 + 운영비율 ≤ 72%**가 필수 조건.
- 기준 시나리오(1,000,000원 주문, 25만원 레퍼럴, 3만원 PG, 원가 50% = 50만원, 운영비 10% = 10만원) → 마진 **120,000원 (12%)**.
- 상위 세대 결손(예: 3대 없음) 시 17%는 플랫폼 귀속으로 마진 개선. 이것은 **기대 레퍼럴 비용은 25% 미만**임을 의미하므로 장기 평균은 더 유리.
- 상품별 차등 원가율에 대응하기 위해 **카테고리별 최저 마진 하한** 정책 병행 권고(관리자 설정).

## 주요 유즈케이스 (Given/When/Then)

### UC-1. 정상 구매 + 3세대 정산
- **Given** A→B→C→D 체인이 존재하고 D가 본인인증을 완료함
- **When** D가 1,000,000원 결제 후 구매확정됨
- **Then** `ReferralLedger`에 `(D.order, C, gen=1, +30,000)`, `(D.order, B, gen=2, +50,000)`, `(D.order, A, gen=3, +170,000)` 3개 원장이 생성된다

### UC-2. 2대까지만 존재 (결손)
- **Given** B→C→D 체인(A 없음)
- **When** D가 1,000,000원 결제·확정
- **Then** 원장 2개만 생성(C=30,000, B=50,000). 17%는 플랫폼 귀속(원장 생성 없음).

### UC-3. 부분 환불
- **Given** UC-1 이후 구매확정 5일 뒤 D가 300,000원 부분 환불(비율 30%)
- **When** 환불 승인
- **Then** `(orderId, C, REVERT, -9,000)`, `(orderId, B, REVERT, -15,000)`, `(orderId, A, REVERT, -51,000)` 3개 원장이 생성되고 `amountNet`에 반영

### UC-4. 셀프레퍼럴 차단
- **Given** 회원가입 폼에 추천코드 입력
- **When** 본인인증 `ci` 가 추천인 `ci`와 동일
- **Then** HTTP 400 `REFERRAL_SELF_FORBIDDEN`, AbuseLog `type=SELF, action=BLOCKED` 기록

### UC-5. 순환참조 차단
- **Given** A→B→C 체인 존재, C가 탈퇴 후 A가 C를 추천인으로 재가입 시도
- **When** 가입 요청
- **Then** HTTP 409 `REFERRAL_CYCLE_DETECTED`, 트랜잭션 롤백

### UC-6. 구매확정 후 7일 경과 후 환불 시도
- **Given** 구매확정 +10일 후 환불 접수
- **When** 관리자가 환불 승인
- **Then** 기본 정책은 **역정산 미수행**, 관리자 수동 확인 시 `REVERT` 가능하며 이미 지급된 경우 `CLAWBACK_REQUESTED` 마킹

### UC-7. 분할 결제
- **Given** 주문에 2개의 `paymentId`(각 500,000원) 연결
- **When** 두 결제 모두 완료 + 구매확정
- **Then** 각 결제 단위로 세대별 원장이 생성되어 총액은 250,000원

### UC-8. 혼합 결제(카드 + 포인트)
- **Given** 총 1,000,000원 중 카드 800,000 + 포인트 200,000
- **When** 구매확정
- **Then** 레퍼럴 기준액 = 800,000원 → 총 지급 200,000원 (24,000 + 40,000 + 136,000)

### UC-9. 미성년자 수취 유보
- **Given** 수익자가 만 19세 미만
- **When** 원장 생성
- **Then** 원장은 `EARN`으로 정상 생성되나 `Payout.status = WITHHELD`, 성년 도달 시 수동 해제

### UC-10. 포트원 결제 실패 복구
- **Given** 결제위젯 호출 후 콜백 누락
- **When** 앱 재진입
- **Then** 서버는 `paymentId`로 포트원 단건 조회 → 실제 상태와 `Order.status` 동기화
