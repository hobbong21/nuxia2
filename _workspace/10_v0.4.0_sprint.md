## 범위 (Must)

### M1. Admin BE API 완성 (backend-engineer)

FE `admin-client.ts`가 기대하는 8 메서드에 일대일 매핑:

| FE 메서드 | BE 엔드포인트 | 응답 |
|-----------|--------------|------|
| `getKpi()` | `GET /admin/kpi` | `{ blockedThisMonth, pendingPayoutKrw, minorHoldCount, activeUserCount }` |
| `getAbuseLogs({kind,cursor,limit})` | `GET /admin/abuse-logs` (이미 존재) | 유지 |
| `getUsers({query,cursor,limit})` | `GET /admin/users` | `{ items: AdminUser[], nextCursor }` |
| `getUser(id)` | `GET /admin/users/:id` | `AdminUser` (role/status/ci mask/createdAt/last activity) |
| `getUserTree(id)` | `GET /admin/users/:id/tree` (이미 존재) | 유지 |
| `getPayouts({cursor,limit})` | `GET /admin/payouts` | `{ items: Payout[], nextCursor }` |
| `flagUser(id, body)` | `POST /admin/users/:id/flag` (이미 존재) | 유지 |
| `releaseMinor(id)` | `POST /admin/users/:id/release-minor` (이미 존재) | 유지 |

**신규 엔드포인트 3개** (`/admin/kpi`, `/admin/users`, `/admin/users/:id`, `/admin/payouts`). **shared-types**에 응답 스키마 추가.

**AdminKpiSchema**:
```ts
export const AdminKpiSchema = z.object({
  blockedThisMonth: z.number().int().nonnegative(),
  pendingPayoutKrw: BigIntStringSchema,
  minorHoldCount: z.number().int().nonnegative(),
  activeUserCount: z.number().int().nonnegative(),
});
```

**AdminUserSchema**:
```ts
export const AdminUserSchema = z.object({
  id: IdSchema,
  email: z.string().email(),
  nickname: z.string(),
  role: UserRoleSchema,
  status: UserStatusSchema,
  ciMasked: z.string(), // "hxxxxxxx" (앞 1자 + x)
  identityVerified: z.boolean(),
  payoutEligibility: z.boolean(),
  flaggedCount: z.number().int().nonnegative(), // AbuseLog에서 해당 유저 관련 개수
  createdAt: IsoDateTimeSchema,
  lastLoginAt: IsoDateTimeSchema.nullable(),
  withdrawnAt: IsoDateTimeSchema.nullable(),
});
```

### M2. JWT 세션 ↔ 관리자 쿠키 통합 (backend + frontend)

- 로그인 시 `accessToken`/`refreshToken` 외에 **관리자 판별용 short-lived cookie 발행**: `Set-Cookie: nx_role=ADMIN; HttpOnly; SameSite=Lax; Secure; Max-Age=3600`
- `apps/web/middleware.ts` 쿠키 가드는 유지, 다만 서버에서 실제 JWT 세션 기반으로 쿠키를 발급하도록 연결
- `apps/web/lib/admin-client.ts` fetch에 `credentials: 'include'` 추가
- 쿠키는 role 정보만, access token은 `Authorization: Bearer` 헤더로 별도 전달 (기존 로직 유지)

### M3. `/metrics` 엔드포인트 + prom-client (backend)

- `apps/api/src/modules/metrics/` 모듈 추가
- `prom-client` devDep 추가 (정확히는 dependency)
- 기본 Node.js 메트릭(default collectors) + custom counters 5개:
  - `nuxia2_referral_distribute_total{result}` (success|skipped|failed)
  - `nuxia2_payment_confirm_total{result}` (success|mismatch|failed)
  - `nuxia2_abuse_blocked_total{kind}` (kind label)
  - `nuxia2_webhook_received_total{source,status}` (source=portone)
  - `nuxia2_minor_hold_total` (gauge)
- `GET /metrics` — 프로덕션에서는 internal-only 가드(`X-Internal-Secret` 헤더 또는 netpol), 개발은 public
- Admin KPI는 metrics에서 집계할 수도 있지만, MVP에서는 DB 직접 쿼리 (복잡도 낮춤)

### M4. Audit Log 서비스 일관화 (backend)

- `apps/api/src/modules/audit/` 모듈
- `AuditLog` 모델은 기존 Prisma 스키마에 이미 있음 (v0.1)
- 서비스 메서드:
  - `log({ actorUserId, kind, targetType, targetId, before?, after?, meta? })`
  - 모든 관리자 API(flag/release-minor/mark-staff/suspend/unsuspend/...)에 interceptor로 자동 주입
- `AuditLogInterceptor` — `@Audit(kind)` 데코레이터로 컨트롤러 메서드 마킹, 응답 성공 시 자동 기록
- Admin API에서 호출하는 민감 변경은 전부 기록

### M5. 2FA (TOTP) UI 준비 (backend + frontend)

**Backend**:
- `User.totpSecret`, `User.totpEnabled` 컬럼은 이미 v0.3 Non-goals에서 스키마만 준비하기로 했으나 실제로는 없음 → **이번에 스키마 추가**
- `POST /auth/2fa/setup` — TOTP secret 생성 + QR 데이터 URI 반환
- `POST /auth/2fa/verify` — 코드 검증 후 `totpEnabled=true`
- `POST /auth/2fa/disable` (2FA 코드 필요)
- `otplib` 또는 `speakeasy` 사용

**Frontend**:
- `/mypage/security` 페이지 — 2FA 활성/비활성 토글
- 활성화 모달: QR 코드 표시 + 6자리 인증번호 입력
- 다음 로그인 시 TOTP 필요 (로그인 페이지에 코드 입력 필드)

### M6. FE admin mock → real API 전환 (frontend)

- `apps/web/lib/admin-client.ts`의 `USE_MOCK` 플래그를 `process.env.NEXT_PUBLIC_USE_MOCK === '1'`로 변경
- BE 미기동 시 로컬 개발에서만 mock
- 실제 API 호출 경로에 zod safeParse 추가
