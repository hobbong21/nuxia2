## 3. Dockerfile 설계 노트

### apps/api (4 stages)

1. `base` — `node:20-alpine`, `corepack enable`, `pnpm@9 activate`
2. `deps` — lockfile 기반 `pnpm install --frozen-lockfile --filter @nuxia2/api... --filter @nuxia2/shared-types`
3. `build` — `prisma generate` → `nest build`. `shared-types` 의 `build`는 `tsc --noEmit` 이므로 실제 dist 가 생성되지 않음 (src/index.ts 이 entrypoint). runtime 단계에서 node_modules/@nuxia2/shared-types 로 전체 워크스페이스 패키지를 복사.
4. `runtime` — non-root (`node2:1001`), `NODE_ENV=production`, `PORT=4000`, HEALTHCHECK on `/health` (TODO: backend-engineer 가 `@nestjs/terminus` 로 엔드포인트 노출).

### apps/web (4 stages)

- `base` / `deps` 구성 동일
- `build` — `next build` (standalone 출력 생성)
- `runtime` — `.next/standalone` 전체 + `.next/static` + `public/` 복사, `node apps/web/server.js` 로 실행. monorepo standalone 특성상 server.js 가 `apps/web/server.js` 경로에 배치됨.
- non-root (`nextjs:1001`), `NEXT_TELEMETRY_DISABLED=1`

### shared-types 처리

- `packages/shared-types` 는 빌드 산출물이 없다 (zod schemas, src 그대로 export).
- runtime 단계에서 `node_modules/@nuxia2/shared-types/` 로 복사해 TS 런타임 없이도 CJS/ESM 엔트리를 찾게 함 — 실제로는 nest/next 가 `transpilePackages` + `ts-node` 로 번들 시점에 흡수하므로 런타임 경로는 안전.

## 4. docker-compose full profile 수정

| 변경점 | 이유 |
|--------|------|
| `env_file` 을 `{ path, required: false }` 로 변경 | `.env.local` 이 CI/Codespaces 에 없을 때도 `compose --profile full up` 가능 |
| `environment:` 블록 추가 | 컨테이너 네트워크 이름 (`postgres`, `redis`, `api`) 을 명시적으로 주입 — localhost 로는 해결되지 않음 |
| `DATABASE_URL`, `REDIS_URL`, `NEXT_PUBLIC_API_BASE_URL` 하드코딩 | 개발 환경에서 바로 `docker compose --profile full up` 성공하도록 |

※ `NEXT_PUBLIC_*` 변수는 원칙적으로 빌드 타임 주입이 필요. 현재는 런타임 fetch 가정 — 빌드 시점에 외부 호스트 주입이 필요해지면 `build.args` 로 승격해야 함.

## 5. CI workflow 설계 (ci.yml)

| Job | 러너 | 서비스 | 주요 단계 |
|-----|------|--------|-----------|
| `typecheck` | ubuntu-latest | 없음 | install → `pnpm -r --if-present run typecheck` |
| `lint` | ubuntu-latest | 없음 | install → `pnpm -r --if-present run lint` |
| `e2e` | ubuntu-latest | postgres:16-alpine, redis:7-alpine | install → `prisma migrate deploy` → `pnpm --filter @nuxia2/api run test:e2e` |

- `concurrency` 그룹으로 브랜치당 최신 푸시만 실행 (자원 절약)
- pnpm cache 활성화 (`actions/setup-node@v4 with cache: pnpm`)
- e2e 테스트용 env 시크릿은 고정 테스트값 (실제 PII 없음, 32자 이상 fulfilled)
- `JWT_SECRET`, `APP_ENCRYPTION_KEY`, `PORTONE_*` — `.env.example` 주석의 길이 요구사항 충족

## 6. docker-build.yml (선택, YES 작성함)

- trigger: `push: [main]` + tag `v*.*.*` + manual
- per-image job: api, web 병렬
- `docker/metadata-action@v5` 로 태그 자동 생성 (branch / semver / sha)
- `docker/build-push-action@v6` + `type=gha` buildx cache
- 플랫폼: linux/amd64 (arm64 는 Won't 로 유보 — 빌드 시간 2배+)
- 레지스트리: `ghcr.io/${owner}/nuxia2-{api,web}`

## 9. 사용법 (다음 단계 검증용 스니펫)

```bash
# 단일 이미지 빌드
docker build -f apps/api/Dockerfile -t nuxia2-api:local .
docker build -f apps/web/Dockerfile -t nuxia2-web:local .

# 전체 스택 (compose full profile)
docker compose --profile full up -d --build

# CI 사전 드라이런
pnpm install --frozen-lockfile
pnpm -r --if-present run typecheck
pnpm -r --if-present run lint
pnpm --filter @nuxia2/api exec prisma migrate deploy
pnpm --filter @nuxia2/api run test:e2e
```
