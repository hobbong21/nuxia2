## 핵심 설계 결정

### compose 구조
- **기본 서비스**: postgres, redis (always)
- **선택 서비스**: api, web (`--profile full` 플래그로만 기동)
  - 로컬 개발은 호스트에서 `pnpm dev` 직접 실행하는 것이 더 빠른 DX
  - profile=full은 통합 검증 또는 CI 시뮬레이션용
- **볼륨 네이밍**: `nuxia2_pg_data`, `nuxia2_redis_data` (prefix로 프로젝트 격리)
- **healthcheck**: pg_isready / redis-cli ping → 5초 간격, 10회 재시도
- **포트 오버라이드**: 환경변수 `${POSTGRES_PORT:-5432}` 패턴 → 호스트 포트 충돌 시 `.env`만 수정

### 초기화 스크립트
- `docker/postgres/init/*.sql`은 **신규 볼륨 초기화 시에만 실행**됨 (docker-entrypoint-initdb.d 관례)
- 기존 볼륨이 있다면 `make reset`으로 초기화해야 SQL 재실행됨
- `pgcrypto`만 활성화. `ltree`는 referral tree 규모에 따라 v0.3+ 검토

### Makefile
- Windows 사용자는 Git Bash 또는 WSL 필요 (또는 `choco install make`)
- help 타겟에 grep/awk 패턴 — Windows Git Bash 동작 확인 필요
- `db-migrate`, `db-seed`는 `pnpm --filter @nuxia2/api` 경로 사용 (v0.1.0 기준 패키지명 보정: `@nuxia/api` → `@nuxia2/api`는 실제 package.json 확인 필요)

## 사용법 예시

### 신규 개발자 온보딩

```bash
git clone https://github.com/hobbong21/nuxia2.git
cd nuxia2
cp .env.docker.example .env
cp apps/api/.env.example apps/api/.env.local
# apps/api/.env.local 편집 (JWT_SECRET 32자 등)

make up          # postgres + redis healthy 도달 (~10초)
pnpm install
make db-migrate  # M1 (backend-engineer) 완료 후 동작
make db-seed
```

### 포트 충돌 해소

호스트에 기존 postgres가 5432를 사용 중이면:
```bash
# .env 수정
POSTGRES_PORT=5433

# apps/api/.env.local 수정
DATABASE_URL=postgresql://nuxia2:nuxia2_dev@localhost:5433/nuxia2?schema=public

make down && make up
```

### 데이터 초기화

```bash
make reset   # 볼륨 삭제 → 재기동 → init SQL 재실행
make db-migrate
make db-seed
```

### 전체 통합 기동 (profile=full)

**전제**: `apps/api/Dockerfile`, `apps/web/Dockerfile` 존재. 본 스프린트에서는 미작성 → TODO.
```bash
make up-full   # 현재는 빌드 실패 예상 (Dockerfile 부재)
```
