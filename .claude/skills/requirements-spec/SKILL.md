# Requirements Spec — 커머스 × 레퍼럴 요구사항 템플릿

커머스 하이브리드앱 + 3세대 레퍼럴 시스템의 요구사항을 일관된 구조로 작성한다.

## 작성 원칙

1. **측정 가능성** — 모든 기능 요구사항은 수치 또는 Given/When/Then으로 검증 가능해야 한다
2. **에지 케이스 선명화** — 정상 케이스 1개당 실패 케이스 2~3개를 병기
3. **정책-검출-차단 3층** — 어뷰징은 "정책 문장"만 쓰지 말고 "검출 방법"과 "차단 액션"까지 함께 쓴다
4. **가정 명시** — 불확실한 가정은 문서 상단 `## Assumptions` 섹션에 모은다

## 출력 문서 구조

```markdown
# 01. 요구사항 명세

## Assumptions
- {가정 1}
- {가정 2}

## 기능 스코프 (MoSCoW)
- **Must**: {필수 기능 목록}
- **Should**: {중요하지만 후순위}
- **Could**: {있으면 좋음}
- **Won't (this release)**: {명시적 제외}

## 사용자 페르소나
| 페르소나 | 목표 | 좌절 지점 |
|---------|------|-----------|
| 일반 구매자 | 빠른 구매 | 복잡한 결제 |
| 레퍼럴 리더 | 조직 확장 | 수익 추적 불투명 |
| 관리자 | 어뷰징 차단 | 수동 모니터링 |

## 레퍼럴 정책 상세표

### 정상 케이스
| 세대 | 비율 | 지급 시점 | 유보 기간 |
|------|------|-----------|-----------|
| 1대 | 3% | 주문 확정(구매확정) | 7일 |
| 2대 | 5% | 동일 | 7일 |
| 3대 | 17% | 동일 | 7일 |

### 환불/취소 케이스
| 이벤트 | 정산 액션 |
|--------|-----------|
| 주문 전체 환불 | 해당 주문의 모든 레퍼럴 역정산 (ledger에 `-` 원장 기록) |
| 부분 환불 | 환불 비율만큼 역정산 |
| 주문 취소(결제 전) | 레퍼럴 원장 생성되지 않음 |
| 구매확정 후 N일 경과 | 역정산 불가 정책 (환불 시 레퍼럴은 보전) |

### 분할 결제 케이스
- 포트원 `paymentId` 단위로 각 부분 결제금액 × 세대 비율 계산
- 총액 = 각 부분 결제의 배분 합

## 어뷰징 시나리오 & 방지 요구사항

### 셀프레퍼럴 (본인이 본인을 추천)
- **정책**: 추천인 체인에 동일 `ci`(본인인증 연계정보) 중복 금지
- **검출**: 회원가입 시 추천인의 `ci`와 신규 회원의 `ci` 일치 여부 실시간 확인
- **차단**: 가입 거부 + 어뷰징 로그 기록

### 순환참조 (A→B→C→A)
- **정책**: 추천인 체인의 ancestor에 자기 자신이 나타나면 안 됨
- **검출**: 가입 트랜잭션 내에서 ancestors 순회(최대 3단계)
- **차단**: 트랜잭션 롤백, 에러 응답

### 다중계정 (동일인이 계정 N개)
- **정책**: 1 `ci` = 1 계정
- **검출**: `ci` 유일성 제약 + IP/디바이스 지문 기반 유사도 점수
- **차단**: 동일 `ci` 재가입 차단 / 점수 임계치 초과 시 수동 심사 플래그

### 봇 구매
- **정책**: 구매는 본인인증 완료 계정만
- **검출**: 이상 패턴(초당 주문 수, 동일 IP 다계정) 모니터링
- **차단**: 레이트 리밋 + 수동 심사

## 데이터 계약 초안

### User
- id, ci(유일), nickname, referrerId(nullable), referralCode(유일), createdAt, status(active|suspended)

### Order
- id, userId, items[], amount, paymentId(포트원), status, createdAt, confirmedAt

### ReferralLedger (원장)
- id, orderId, beneficiaryUserId, generation(1|2|3), amount, type(EARN|REVERT), createdAt
- UNIQUE(orderId, beneficiaryUserId, generation)

### Payout (정산)
- id, userId, periodStart, periodEnd, amount, status(PENDING|PAID|WITHHELD)

## 마진 검증 모델

```
총 매출 100% = 상품원가 X% + 플랫폼 마진 Y% + 레퍼럴 지급 25% + 결제수수료 약 3%
→ Y ≥ 운영비 + 수익 목표
```

플랫폼 마진 Y가 운영비 + 목표 수익을 감당 가능한지 숫자로 검증.

## 합격 기준 (qa-integrator가 재사용)

- [ ] 3세대 초과 추천인에는 지급되지 않음
- [ ] 주문 확정 시 `ReferralLedger`에 최대 3개 원장 생성
- [ ] 환불 시 동일 수량의 `REVERT` 원장 생성, 합산 순액 = 0
- [ ] 셀프레퍼럴 회원가입 차단
- [ ] 순환참조 트랜잭션 롤백
- [ ] 1,000,000원 주문 기준 C/B/A에게 각 30,000 / 50,000 / 170,000원 기록
```
